# Phase 4.fem.eig.3.5.4 — implementation plan

**Spec:** `docs/superpowers/specs/2026-05-20-phase-4-fem-eig-3-5-4-design.md`
**Base SHA:** `84a92e7` (post Phase 4.fem.eig.3.5.3 merge)
**Lane:** `crates/yee-fem/**`, `crates/yee-validation/**`,
`docs/src/tutorials/07-fem-open-cavity.md`, `ROADMAP.md`. The yee-py
binding shape (Python wave-port multi-mode kwarg) is **out of lane**
in v3.5.4 — defer to v3.5.4.1 once the Rust API stabilises.

## Step ladder

### M1 — `PortMode` extraction + `PortDefinition` rewrap

`crates/yee-fem/src/open_boundary.rs`:

1. Define `pub struct PortMode { beta_mode, modal_e_t, a_inc: Complex64 }`.
2. Rewrap `PortDefinition` as `{ modes: Vec<PortMode> }`.
3. `impl Debug` per existing pattern.
4. Add a constructor convenience
   `PortDefinition::single_mode(beta_mode, modal_e_t) -> Self` that
   collapses to `vec![PortMode { ..., a_inc: Complex64::ONE }]` so
   v3.5.3 / fem-eig-004 / fem-eig-005 call sites stay one-line.

**Verification:** `cargo check --workspace` exit 0.

### M2 — Assembly path multi-mode summation

`crates/yee-fem/src/element/port.rs`:

1. `assemble_port_face_block` loops over `port.modes`, accumulates
   the stiffness contribution per mode.
2. `assemble_port_modal_rhs` loops over `port.modes`, scales each
   `b_port^{p,m}` by `mode.a_inc`, sums into the global RHS.

`crates/yee-fem/src/open_boundary.rs::solve_at_frequency`:

3. Post-solve `S_{p,p}` extraction picks the mode with `a_inc != 0`
   as the driving mode `m₀`; computes
   `S_{p, m₀} = ⟨E_h, e_t^{p, m₀}⟩ / ⟨e_t^{p, m₀}, e_t^{p, m₀}⟩`.
4. If multiple modes carry `a_inc != 0` (currently not used but
   reserved for v3.5.5 dual-feed): error out with a clear
   `MultipleDrivingModes` variant.

**Verification:** `cargo test --release -p yee-fem` exit 0. The
v3.5.3 single-mode tests (fem-eig-004 / fem-eig-005 / fem-eig-006
no-NaN) stay green by construction since `single_mode` constructor
preserves their call shape.

### M3 — fem-eig-006 driver multi-mode +x port

`crates/yee-validation/src/lib.rs::run_fem_eig_006_high_aspect_pml`:

1. Add helpers `fem_eig_006_beta_te20`, `fem_eig_006_modal_e_t_te20`,
   `fem_eig_006_beta_te01`, `fem_eig_006_modal_e_t_te01`. The TE_{mn}
   field pattern on a rectangular port face (broad-wall `a` along
   `x`, narrow-wall `b` along `y`, propagation along `+z` for the
   purposes of the port-local frame; cavity orientation translates
   to the +x face of the high-aspect cavity) is

   ```
   E_y^{m,n}(x, y) = sin(m·π·x/a) · cos(n·π·y/b)    (TE_{mn})
   E_x^{m,n}(x, y) = (m / n) · (b / a) · cos(...) · sin(...)
   ```

   (Components written into the world-frame tangential basis at the
   +x face per Pozar §3.3; the world-frame mapping mirrors
   the existing `fem_eig_006_modal_e_t_te10` orientation.)

2. Construct the +x `PortDefinition` as a 3-mode vector
   `[TE_{10} (a_inc=1), TE_{20} (a_inc=0), TE_{01} (a_inc=0)]`.

**Verification:**
`cargo test --release -p yee-validation --test fem_eig_006_high_aspect_pml`
exit 0 on default path (smoke + no-NaN). Then run
`-- --include-ignored` to measure the strict gate.

### M4 — Strict-gate disposition

**If** `|S_{11}|(30 GHz) < 0.1`:

1. Remove `#[ignore]` from `fem_eig_006_magnitude_bounded`.
2. Replace the docstring with a "Phase 4.fem.eig.3.5.4 retire"
   subsection citing the measurement and the multi-mode termination.

**Otherwise** (escape-hatch):

1. Keep `#[ignore]`; update docstring with the v3.5.4 multi-mode
   measurement and the next-step queue (v3.5.5 four-mode + TE_{11},
   or v3.5.6 absorbing-mode wave-port).

**Verification:**
`cargo test --release -p yee-validation --test fem_eig_006_high_aspect_pml`
exit 0 (default path; ignored gate either retired or kept ignored
per the branch taken above).

2. `ROADMAP.md`: new Phase 4.fem.eig.3.5.4 line under the
   open-boundary chain, citing the v3.5.4 spec + plan + commits +
   merge SHA. If gate retires, mark the fem-eig-006 line closed.

3. ADR-0047 (`docs/src/decisions/0047-phase-4-fem-eig-3-5-4-multi-mode-wave-port.md`)
   recording: "Decided to extend `PortDefinition` to
   `Vec<PortMode>` rather than introduce a parallel
   `MultiModeWavePort` type, because (i) every wave-port is
   structurally a modal basis projection and the single-mode case
   is a degenerate trivial vector, and (ii) future absorbing-mode
   ports (v3.5.6) plug directly into the same `Vec<PortMode>`
   slot."

## Escape-hatch

If after M4 the multi-mode gate is still `≥ 0.1`, **do not bump
the API surface again in v3.5.4**. Commit M1+M2+M3 as a Rust-side
API extension that any future caller can use; leave the
fem-eig-006 gate `#[ignore]`'d; document the new measurement
faithfully; queue v3.5.5 (four-mode addition of TE_{11} per Pozar
§3.3) and v3.5.6 (absorbing-mode wave-port per Lee-Mittra
1997 §IV) in ADR-0047.

* CLI multi-mode wave-port spec ingestion.
