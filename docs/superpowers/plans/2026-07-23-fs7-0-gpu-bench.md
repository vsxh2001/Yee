# FS.7.0 implementation plan — GPU bench + first optimization passes

Spec: `docs/superpowers/specs/2026-07-23-fs7-0-gpu-bench-design.md`
Branch: `feature/fs7.0-gpu-bench` (from `main` @ post-78cd12f)
Lane: `crates/yee-compute/**` only, plus Task 4's two doc files.

## Global constraints (bind every task)

- **Machine has a real GPU** (RTX 5060 Ti). Every GPU test/gate you run must print
  `adapter 'NVIDIA GeForce RTX 5060 Ti'` — a SKIPPED/NoAdapter result is a task
  failure, not a pass.
- **Never weaken an existing assertion or tolerance.** Bit-exact gates
  (`graded_uniform_bitexact`, compute-018, compute-020 in `gpu_graded_parity`) and
  parity gates (`gpu_cpu_parity` et al.) must pass unmodified.
- Perf numbers are honest measurements. If a change doesn't help, revert it and
  record the negative result. The 3405 Mcells/s bar may be unreachable on a
  448 GB/s card — document, don't fake.
- `cargo clippy -p yee-compute --all-targets -- -D warnings` and
  `cargo fmt --check` clean before every commit. `#![warn(missing_docs)]` holds:
  new public items documented.
- Commit style: `yee-compute: <subject>` ≤72 chars, body says why. Commit per task.
- Benchmark timing methodology (from the T4 session finding): warm-up steps, then
  time `step_n` + `GpuFdtd::sync()`; readback excluded from the stepping figure.

## Task 1 — `GpuFdtd::sync()` + reproducible bench example

- Add `pub fn sync(&self)` to `GpuFdtd` (`crates/yee-compute/src/gpu.rs`): submit an
  empty encoder + block on `device.poll(wgpu::PollType::Wait)` (wgpu 29 API — check
  what the crate's existing readback path uses and reuse that idiom; `read_fields`
  already blocks somewhere — extract/share that wait).
- Add `crates/yee-compute/examples/bench.rs`:
  - grids {64,96,128,160,192,224}³ vacuum (`FdtdSpec::vacuum`), `Fields::with_gaussian_ez`.
  - per grid: CPU (`CpuFdtd`) and GPU (`GpuFdtd`) — warm-up 10 steps + sync, then
    3 reps of (200 steps + sync), median; Mcells·step/s; GPU also: one timed
    `read_fields` (readback ms) and estimated GB/s (report bytes/cell·step assumption inline).
  - GPU absent → print SKIP and exit 0 (NoAdapter idiom). GPU alloc fail at a size →
    print n/a for that size, continue (no panic; note: wgpu validation errors abort
    via uncaptured handler — construct per-size in a fresh scope; if the panic
    can't be caught, probe sizes in a child process or accept process-level skip and
    document).
  - 64³ sanity: CPU-vs-GPU per-E-component rel L2 ≤ 1e-3 → prints PASS/FAIL, exit 1 on FAIL.
  - `--json` flag → machine-readable results on stdout instead of the table.
- Pattern files: `crates/yee-compute/tests/gpu_cpu_parity.rs` (norms + skip idiom).
- Verify: `cargo run -p yee-compute --release --example bench` on this machine
  produces the table with real numbers; `cargo test -p yee-compute --release --test
  gpu_cpu_parity` still green.

## Task 2 — workgroup-shape tuning

- `crates/yee-compute/src/shaders/fdtd.wgsl`: the 9 volume kernels use
  `@workgroup_size(4,4,4)` with (presumably) gid→(i,j,k) direct mapping; x is the
  contiguous axis. Experiment: at least {(64,1,1), (32,2,2), (8,8,4)} vs baseline
  (4,4,4) — via a small local edit loop (sed or override constants), measuring with
  Task 1's bench at 128³ and 192³.
- Keep the single best shape hardcoded (no runtime knob — YAGNI); update the
  matching `dispatch_workgroups` math in `gpu.rs`.
- Record ALL shapes × numbers in the task report (the negative results are the
  documentation).
- Verify: full yee-compute release suite incl. `--include-ignored` GPU gates green
  on hardware; `gpu_graded_parity` bit-exact sub-gate specifically green.

## Task 3 — 192³ throughput dip root-cause

- Reproduce with Task 1 bench (192³ ≈ 1.4 vs 128³ ≈ 2.5 Gcells/s after `78cd12f`).
- Hypotheses to test in order: (a) `STEPS_PER_SUBMIT` chunking (sweep the constant:
  measure 192³ at e.g. 8/32/64/128 steps-per-submit); (b) per-pass bind-group /
  encoder overhead scaling; (c) >128 MiB single-binding slow path in the driver
  (split fields arena? measure-only — do NOT restructure buffers in this increment);
  (d) power/thermal (check `nvidia-smi dmon` during run).
- Fix what's software and small (e.g. a better STEPS_PER_SUBMIT — possibly
  size-dependent); document what isn't. A dip explained-but-not-fixed is an
  acceptable outcome if the explanation is measured, not guessed.
- Verify: bench table before/after in the report; suite green.
