# ADR-0003: Build Python wheels as `abi3-py310`

**Status:** Accepted
**Date:** 2026-05-16
**Deciders:** Yee maintainers

## Context

`yee-py` exposes a Python binding for the Yee solver via PyO3. The
shipping channel is wheels built with `maturin` and uploaded to PyPI
(and to a project-controlled package index for nightly artefacts).

CPython's C API has two stability tiers:

- The **version-specific ABI** changes between minor releases. A
  binding compiled against CPython 3.11 will not load into 3.12, will
  not load into 3.13, and so on. Distributors must publish one wheel
  per (CPython minor, OS, arch) cell, so a five-version matrix on three
  platforms is fifteen wheels per release.
- The **limited / stable ABI** (PEP 384, available since CPython 3.2)
  exposes a curated subset of the C API marked as ABI-stable. A binding
  compiled with `Py_LIMITED_API` set to a baseline version (e.g.
  `0x030A0000` for 3.10) runs on that CPython minor and every
  subsequent minor without recompilation.

PyO3 turns the limited-ABI build path on via the `abi3-pyXY` family of
crate features. With `pyo3 = { version = "0.28", features =
["abi3-py310"] }`, the resulting `.so` / `.pyd` is named with the
`abi3` tag (`yee_py.abi3.so` on Linux, not `yee_py.cpython-311-x86_64-
linux-gnu.so`) and is loadable from any CPython 3.10 or newer.

The trade-off is real but small:

- A handful of PyO3 conveniences (most prominently buffer-protocol
  micro-optimisations and some free-list short-circuits) require the
  full version-specific ABI. The performance cost in measurements
  collected during Phase 1.py.0 was within the noise floor of
  end-to-end solve time (the dense LU dominates).
- Some upstream CPython features arrive on the stable ABI several
  releases after they appear on the version-specific ABI. None of
  these are on Yee's near-term roadmap.

The alternative — shipping one wheel per CPython minor — would inflate
release artefact count and CI minutes, and would mean a user landing on
a brand-new CPython release waits days or weeks for a Yee build that
matches their interpreter.

A second axis is **free-threaded Python**, the PEP 703 build of CPython
that disables the Global Interpreter Lock. Free-threaded builds exist
as `python3.13t`, `python3.14t`, etc. They are NOT covered by the
current `Py_LIMITED_API`/`abi3` machinery — a free-threaded interpreter
needs C extensions built against `python3.13t`'s headers with a separate
ABI tag. PEP 803 (in draft as of 2026-Q2) proposes extending the stable
ABI to cover free-threaded builds, at which point Yee can collapse the
free-threaded matrix back into a single wheel.

The lower bound choice is **3.10**:

- 3.10 was released October 2021. By May 2026 it is well past the
  community uptake curve.
- 3.9 reached end-of-life on 2025-10-31 per PEP 596 / status update.
  Dropping 3.9 is conventional for a 2026-shipped library.
- 3.10 introduced structural pattern matching, parenthesised context
  managers, and the `int | None` union shorthand — features the Yee
  Python user-facing API leans on in type stubs and examples.

## Decision

Build `yee-py` wheels with PyO3's `abi3-py310` feature enabled.

`yee-py/Cargo.toml`:

```toml
[dependencies]
pyo3 = { version = "0.28", features = ["extension-module", "abi3-py310"] }
```

`yee-py/pyproject.toml` (consumed by maturin):

```toml
[tool.maturin]
features = ["pyo3/extension-module", "pyo3/abi3-py310"]
python-source = "python"
strip = true
```

Build matrix:

- **Linux** — `manylinux_2_28` x86_64 and aarch64, single wheel each.
- **macOS** — `macosx_11_0` x86_64 and arm64, single wheel each
  (universal2 considered and rejected; doubles wheel size for limited
  user benefit on a numerics package).
- **Windows** — `win_amd64`, single wheel.

That is **five wheels per release**, covering Python 3.10 through
whatever is the current CPython minor at release time (currently
3.14).

Supported Python versions: **3.10, 3.11, 3.12, 3.13, 3.14** (all on the
stable ABI). Python 3.9 and earlier are explicitly not supported.

Free-threaded CPython (3.13t, 3.14t) is **not supported in this ADR**;
a separate ADR will revisit this once PEP 803 ships or once `pyo3`
gains a free-threaded-stable-ABI feature flag.

## Consequences

**What becomes easier:**

- Single-wheel distribution per (OS, arch). Five wheels, not 25. CI
  build time drops accordingly.
- New CPython minor releases (3.15, 3.16, ...) are automatically
  supported on day one without a Yee re-release.
- Distro packagers building Yee against the system Python don't have
  to know which minor the user is on; the same `.so` works.

**What becomes harder:**

- Free-threaded Python (`python3.13t`, `python3.14t`) is NOT covered
  by current `abi3-py310`. Users on free-threaded builds will get an
  `ImportError` from `import yee` and need to install standard CPython
  to use Yee. Until PEP 803 lands this is a hard limitation.
- A few `pyo3` APIs are unavailable under `abi3` — most notably some
  `PyClass` specialisations that depend on private CPython structures.
  This has not bitten Yee's API surface yet but is a constraint on
  future binding design.
- `pyo3` itself occasionally finds bugs in CPython's stable ABI when a
  new minor ships; we may need to land a `pyo3` patch update faster
  than we would for a version-specific build.

**What's now closed off:**

- Supporting CPython 3.9 or earlier is closed off. The `abi3-py39`
  feature exists in `pyo3` but would force us to forgo language
  features that the Python user-facing API documentation already
  assumes (pattern matching in examples, `X | None` type hints).
- Shipping per-minor wheels for performance reasons. The maintainers
  have measured the gap and judged it not worth the matrix explosion.
